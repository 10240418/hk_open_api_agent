# HK Knowledge Base Runtime Instructions

You are operating inside the HK company project knowledge base. Use this repository as the primary context for questions about HK projects, deployment, domains, ports, server operations, and service reconstruction.

## Repository Map

- `README.md`: entry point and operating rules.
- `inventory/projects.yaml`: structured source of truth for all known Git projects.
- `inventory/service-map.md`: business/service overview.
- `inventory/servers.yaml`: server inventory.
- `inventory/domains.yaml`: domain inventory.
- `inventory/ports.yaml`: port inventory.
- `inventory/env-files.md`: non-secret index of environment files and where sensitive backups exist.
- `projects/*.md`: one knowledge document per project.
- `runbooks/*.md`: deployment, rebuild, verification, and update procedures.
- `ai/deployment-agent-rules.md`: required deployment behavior.
- `ai/knowledge-update-rules.md`: required knowledge maintenance behavior.
- `deployments/`: deployment records to create or update after successful deployments.

## Required Workflow

When the user asks about a project, service, domain, port, deployment, or migration:

1. First inspect `inventory/projects.yaml` and `inventory/service-map.md`.
2. Then inspect the relevant `projects/*.md` file.
3. For deployment or rollback work, also inspect `ai/deployment-agent-rules.md` and the relevant `runbooks/*.md`.
4. If the user provides only a domain, search `inventory/domains.yaml`, `inventory/ports.yaml`, and `projects/*.md`.
5. If the user provides only a project name, match it against `inventory/projects.yaml` and `projects/*.md`.
6. Before changing a server, produce a concise plan including target host, files to modify, commands to run, verification steps, and rollback path.
7. After a successful deployment or meaningful infrastructure change, create or update a file under `deployments/` and update the related inventory or project document.

## Safety Rules

- Never read, print, commit, or send secrets from `.env`, `.env.*`, or `private/` unless the user explicitly asks for a local/server configuration operation that requires them.
- If a secret is needed, reference the variable name and location, not the value.
- Do not expose API keys, SSH keys, database passwords, OAuth tokens, QQ secrets, MiMo keys, DeepSeek keys, or environment backup contents in chat.
- Treat `inventory/*.yaml` as more authoritative than prose Markdown if they conflict.
- Ask for confirmation before destructive actions, DNS changes, database migrations, or production restarts unless the user has explicitly authorized the exact operation.

## Preferred Answer Style

- Answer in Chinese by default.
- Be direct and operational.
- For deployment tasks, include exact commands only after checking the relevant runbook and project document.
- When information is missing, state what file should be updated so future runs become more automatic.
